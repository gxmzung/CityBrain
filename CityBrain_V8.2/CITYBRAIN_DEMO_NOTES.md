# CityBrain Demo Notes

## 1. One-line Summary
CityBrain은 학생식당의 메뉴, 혼잡 체감, 운영 입력, 학생 반응 데이터를 연결해 학생과 운영자 모두의 의사결정을 줄여주는 스마트캠퍼스 식당 MVP이다.

## 2. Problem
학생 입장에서는 오늘 메뉴, 혼잡도, 인기 메뉴 소진 여부, 식당 이용 타이밍을 미리 알기 어렵다.

운영 입장에서는 학생 수요를 감으로 판단하는 경우가 많고, 메뉴 반응이나 혼잡 체감 데이터를 운영 개선에 바로 활용하기 어렵다.

## 3. Core User
- 학생: 오늘 어디서, 언제, 무엇을 먹을지 빠르게 결정하고 싶은 사람
- 식당 운영자: 최소 입력으로 메뉴/혼잡/반응 데이터를 관리하고 싶은 사람
- 학교/심사위원: 스마트캠퍼스 관점에서 실제 운영 가능성과 확장성을 보고 싶은 사람

## 4. MVP Scope
- 학생용 화면
  - 오늘 메뉴 확인
  - 혼잡도/체감 정보 확인
  - 메뉴 반응 또는 만족도 입력

- 관리자용 화면
  - 메뉴 등록/수정
  - 혼잡도 또는 운영 상태 입력
  - 학생 반응 요약 확인

- 백엔드
  - FastAPI 기반 API
  - SQLite 기반 데이터 저장
  - 학생 웹/관리자 웹/Android 앱 연동 가능 구조

## 5. Demo Flow
1. 학생이 앱 또는 웹에서 오늘 메뉴를 확인한다.
2. 혼잡도 또는 식당 상태를 보고 방문 타이밍을 판단한다.
3. 식사 후 메뉴 반응을 남긴다.
4. 운영자는 관리자 화면에서 메뉴와 상태를 최소 입력으로 관리한다.
5. 학생 반응과 운영 데이터를 통해 다음 운영 개선의 근거를 만든다.

## 6. Judge Q&A Ready Points

### Q1. 기존 학교 앱과 뭐가 다른가?
기존 학교 앱이 공지와 정보 전달 중심이라면, CityBrain은 식당이라는 구체적인 생활 문제를 중심으로 학생 체감 데이터와 운영 데이터를 연결한다.

### Q2. 혼잡도는 어떻게 믿을 수 있나?
초기 MVP에서는 운영자 입력 또는 학생 체감 입력 기반으로 시작한다. 이후 키오스크 로그, 결제 건수, 시간대별 방문 패턴과 연동하면 예측형 모델로 확장할 수 있다.

### Q3. 운영자에게 부담이 되지 않나?
핵심은 직원 입력을 늘리는 것이 아니라 최소 입력 UI를 제공하는 것이다. 메뉴, 혼잡 상태, 소진 여부처럼 운영 판단에 필요한 항목만 빠르게 입력하도록 설계한다.

### Q4. 개인정보 문제는 없나?
초기 MVP는 개인 식별 정보 없이 메뉴 반응, 시간대, 혼잡 체감 같은 비식별 운영 데이터 중심으로 설계한다. 학생 인증이 필요한 경우에도 최소 수집 원칙을 적용해야 한다.

### Q5. 실제 도입 가능성은?
전체 학교 시스템 교체가 아니라 학생식당이라는 제한된 영역에서 시범 운영할 수 있다. 따라서 초기 도입 부담이 낮고, 운영 효과를 확인한 뒤 다른 시설로 확장할 수 있다.

## 7. Honest Limits
- 현재 MVP는 실제 키오스크/결제 시스템과 완전 연동된 상태는 아니다.
- 혼잡도는 초기에는 예측보다 입력/체감 기반에 가깝다.
- 개인정보 처리방침, 로그인/학생 인증, 장애 대응 정책은 정식 서비스 전에 보강해야 한다.
- 앱스토어 배포 수준으로 가려면 접근성, 반응형 UI, 보안, 운영 로그, 백업 정책이 추가로 필요하다.
