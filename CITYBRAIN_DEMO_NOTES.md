# CityBrain Demo Notes

## Purpose

이 문서는 CityBrain의 심사 발표와 시연을 준비하기 위한 데모 노트이다.

목표는 CityBrain을 단순한 학생식당 메뉴 앱이 아니라, 학생의 식사 선택과 식당 운영 개선을 연결하는 스마트캠퍼스 MVP로 설명하는 것이다.

---

## 1. One-line Summary

CityBrain은 학생식당 정보를 단순히 보여주는 앱이 아니라, 학생의 식사 선택과 식당 운영 개선을 연결하는 스마트캠퍼스 서비스이다.

학생은 메뉴, 혼잡도, 남은 수량, 만족도 정보를 확인하고,  
학교와 식당은 학생 반응과 운영 데이터를 바탕으로 식수 예측, 메뉴 수량 조절, 혼잡 분산, 민원 감소에 활용할 수 있다.

---

## 2. Problem

학생 입장에서는 오늘 메뉴, 현재 혼잡도, 인기 메뉴 소진 여부, 식당 이용 타이밍을 미리 알기 어렵다.

그래서 학생은 직접 식당에 가보고 줄이 길어서 포기하거나, 먹고 싶은 메뉴가 이미 소진되어 헛걸음하는 상황을 겪을 수 있다.

식당 운영자 입장에서도 학생들이 어떤 메뉴를 선호하는지, 어느 시간대에 몰리는지, 어떤 불만이 반복되는지 데이터로 보기 어렵다.

결국 학생은 불편을 겪고, 운영자는 감과 경험에 의존해 식당을 운영하게 된다.

---

## 3. Core Users

### Student

학생은 식당에 직접 가보기 전에 아래 정보를 확인하고 싶어 한다.

```text
1. 오늘 메뉴
2. 현재 혼잡도
3. 인기 메뉴 남은 수량 또는 소진 여부
4. 다른 학생들의 만족도 반응
5. 식당 방문 타이밍
```

### Operator

식당 운영자는 최소한의 입력으로 아래 정보를 관리하고 싶어 한다.

```text
1. 오늘 메뉴
2. 메뉴 상태
3. 혼잡도
4. 소진 여부
5. 학생 만족도와 반응
```

### School / Judge

학교와 심사위원은 단순 편의 기능보다 아래 가치를 확인하고 싶어 한다.

```text
1. 실제 문제인지
2. 도입 가능성이 있는지
3. 운영 데이터가 남는지
4. 학생 복지와 행정 효율에 도움이 되는지
5. 작은 범위에서 시범운영 가능한지
```

## 4. MVP Scope

### Student Side

```text
1. 오늘 메뉴 확인
2. 혼잡도 또는 식당 상태 확인
3. 인기 메뉴 남은 수량 또는 소진 상태 확인
4. 메뉴 만족도 또는 반응 입력
```

### Admin Side

```text
1. 메뉴 등록
2. 메뉴 수정
3. 혼잡도 입력
4. 소진 상태 입력
5. 학생 반응 확인
```

### Backend

```text
1. FastAPI 기반 API
2. SQLite 기반 데이터 저장
3. 학생 웹 / 관리자 웹 / Android 앱 연동 가능 구조
```

## 5. 90-Second Demo Flow

### 90초 시연 흐름

```text
1. 학생 화면 접속
```

### 90초 발표 대본

CityBrain은 학생식당 정보를 단순히 보여주는 앱이 아니라, 학생의 식사 선택과 식당 운영 개선을 연결하는 스마트캠퍼스 서비스입니다.

학생은 앱에서 오늘 메뉴, 현재 혼잡도, 인기 메뉴의 남은 수량, 만족도 반응을 확인할 수 있습니다. 그래서 직접 식당에 가보고 줄이 길거나, 먹고 싶은 메뉴가 이미 소진돼서 헛걸음하는 상황을 줄일 수 있습니다.

운영자 입장에서도 장점이 있습니다. 메뉴 상태, 혼잡도, 소진 여부를 최소 입력으로 관리하고, 학생들이 남긴 만족도와 반응을 확인할 수 있습니다.

학교 입장에서 CityBrain의 핵심 가치는 단순한 이미지 개선이 아니라 운영 데이터입니다. 메뉴 만족도, 혼잡 시간대, 소진 불만, 학생 반응 데이터가 누적되면 식수 예측, 메뉴 수량 조절, 혼잡 분산, 민원 감소에 활용할 수 있습니다.

즉, CityBrain은 학생에게는 더 나은 식사 선택을 제공하고, 학교와 식당에는 데이터 기반 운영 개선의 근거를 제공하는 스마트캠퍼스 MVP입니다.

## 7. School-side ROI

### Core ROI Sentence

```text
CityBrain의 학교 측 ROI는 이미지가 아니라 운영 데이터다.
```

### Explanation

학교가 CityBrain을 통해 얻는 가장 큰 이득은 학생식당 운영 데이터를 확보할 수 있다는 점이다.

학생 편의만 보면 단순 복지 앱처럼 보일 수 있지만, CityBrain은 메뉴 만족도, 혼잡 시간대, 인기 메뉴 소진 여부, 반복 불만 데이터를 누적할 수 있다.

이 데이터는 식당 운영자가 감이 아니라 근거를 가지고 운영을 조정하는 데 활용될 수 있다.

### School-side Benefits

```text
1. 학생식당 운영 데이터 확보
2. 메뉴별 만족도 파악
3. 시간대별 혼잡 패턴 확인
4. 인기 메뉴 소진 빈도 확인
5. 식수 예측과 메뉴 수량 조절 근거 확보
6. 반복 민원 사전 감지
7. 학생 복지 만족도 향상
8. 스마트캠퍼스 시범운영 성과 확보
```

### Presentation Version

학교 입장에서 CityBrain의 가치는 단순히 학생 편의를 높이는 데서 끝나지 않습니다.

메뉴 만족도, 혼잡 시간대, 인기 메뉴 소진 여부, 반복 불만 데이터를 누적하면 식당 운영을 감이 아니라 데이터 기반으로 개선할 수 있습니다.

이를 통해 식수 예측, 메뉴 수량 조절, 혼잡 분산, 민원 감소에 활용할 수 있고, 학교는 학생식당이라는 제한된 범위에서 스마트캠퍼스 시범운영 성과를 확인할 수 있습니다.

## 8. Judge Q&A

### Q1. 기존 학교 앱과 뭐가 다른가?

기존 학교 앱은 공지와 정보 전달 중심이다.

CityBrain은 학생식당이라는 구체적인 생활 문제를 중심으로 메뉴, 혼잡도, 만족도, 소진 데이터를 연결해 운영 개선까지 이어지는 서비스이다.

---

### Q2. 혼잡도나 남은 수량은 어떻게 믿을 수 있는가?

초기 MVP에서는 운영자 입력과 학생 체감 입력으로 시작한다.

이후에는 키오스크 로그, 결제 건수, 시간대별 방문 패턴과 연동해 신뢰도를 높일 수 있다.

---

### Q3. 직원 입력 부담이 커지지 않는가?

핵심은 많은 입력이 아니라 최소 입력이다.

메뉴 상태, 혼잡도, 소진 여부처럼 운영에 필요한 항목만 빠르게 입력하도록 설계한다.

---

### Q4. 학교가 얻는 이득은 무엇인가?

가장 큰 이득은 운영 데이터다.

메뉴 만족도, 혼잡 시간대, 소진 불만 데이터를 통해 식수 예측, 수량 조절, 혼잡 분산, 민원 감소에 활용할 수 있다.

---

### Q5. 개인정보 문제는 없는가?

초기 MVP는 개인 식별 정보 없이 메뉴 반응, 시간대, 혼잡 체감 같은 비식별 운영 데이터를 중심으로 설계한다.

학생 인증이 필요해도 최소 수집 원칙을 적용해야 한다.

---

### Q6. 실제 도입 가능성이 있는가?

학교 전체 시스템을 처음부터 바꾸는 방식이 아니라 학생식당이라는 제한된 영역에서 시범운영하는 방식이다.

그래서 도입 부담이 낮고, 효과가 확인되면 다른 시설로 확장할 수 있다.

---

## 9. Current Limitations

현재 MVP는 심사와 시범운영 가능성 검증을 위한 단계이며, 정식 서비스 수준으로 가기 전에는 아래 보완이 필요하다.

```text
1. 실제 키오스크 / 결제 시스템과 완전 연동된 상태는 아니다.
2. 혼잡도와 남은 수량은 초기에는 운영자 입력 또는 학생 체감 입력 기반이다.
3. 로그인과 학생 인증 기능은 정식 서비스 전에 보강해야 한다.
4. 개인정보 처리방침과 데이터 보관 정책이 필요하다.
5. 장애 대응, 운영 로그, 백업 정책이 아직 부족하다.
6. Android 앱은 시연 환경의 서버 IP와 네트워크 설정에 영향을 받는다.
7. 접근성, 반응형 UI, 앱스토어 배포 정책은 추가 검토가 필요하다.
8. 실제 식당 운영자 인터뷰와 사용자 테스트를 통해 입력 부담을 더 검증해야 한다.
```

### Honest Limitation Statement

현재 CityBrain은 실제 운영 시스템을 완전히 대체하는 단계가 아니라, 학생식당이라는 제한된 영역에서 문제와 가능성을 검증하는 MVP입니다.

혼잡도와 남은 수량은 초기에는 운영자 입력과 학생 체감 데이터를 기반으로 시작하고, 이후 키오스크 로그나 결제 건수와 연동하면 신뢰도를 높일 수 있습니다.

정식 서비스로 확장하려면 개인정보 처리방침, 로그인/학생 인증, 장애 대응, 운영 로그, 백업 정책을 추가로 보강해야 합니다.
